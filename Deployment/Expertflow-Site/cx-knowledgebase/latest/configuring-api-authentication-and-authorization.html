Configuring API Authentication and Authorization

This configuration is valid for CX5.1.0 and onwards!

This guide explains how to enable or disable API Authentication (verifying who is calling your API) and API Authorization (controlling what they are allowed to do) in your CX deployment using Application Gateway and IAM.

  • API authentication is the process of validating an entity's credentials to prove they are who they claim to be, granting entry to the system.

  • API authorization is the process of controlling access to specific resources and actions based on the permissions granted to an authenticated entity. 

Before You Start

You need the following values before you apply any configurations for API security.

  • <yourRealm>: Your tenant ID or realm name (e.g., expertflow) used in IAM.

Steps to Configure Authentication & Authorization

  1. On your server where CX is deployed, go to the folder that contains deployment files.

  2. Navigate to /kubernetes/helm-values.

  3. Inside this folder, you’ll find a file named apisix-custom-values.yaml.

  4. To enable or disable the authentication and authorization, make the value of global.oidc.enabled as true (enable) or false (disable). It will look something like this by default. Note: Please note that it will enable or disable both authentication and authorization on all the endpoints mentioned in this file using this single flag.

    global:
  ingressRouter: "*.expertflow.com"
  ingressClassName: "nginx"
  ingressTlsCertName: "ef-ingress-tls-secret"

  # Open ID Connect and Authz Configuraiton - Enabling or Disabling AUTH
  keycloak:
    realm: "<yourRealm>" # By default it will be "expertflow" otherwise you can update it accordingly
    clientId: "cim"
    clientSecret: "ef61df80-061c-4c29-b9ac-387e6bf67052"
  oidc:
    enabled: false
    discovery: "http://keycloak.ef-external.svc/auth/realms/{{ .Values.global.keycloak.realm }}/.well-known/openid-configuration"
    realm: "{{ .Values.global.keycloak.realm }}"
    clientId: "{{ .Values.global.keycloak.clientId }}"
    clientSecret: "{{ .Values.global.keycloak.clientSecret }}"
    authz:
      discovery: "http://keycloak.ef-external.svc/auth/realms/{{ .Values.global.keycloak.realm }}/.well-known/uma2-configuration"
      resourceRegistrationEndpoint: "http://keycloak.ef-external.svc/auth/realms/{{ .Values.global.keycloak.realm }}/authz/protection/resource_set"

  5. After enabling or disabling a plugin, exit /helm-values directory and go back to /kubernetes directory. Run the following command:

    helm upgrade --install --namespace ef-external --values helm-values/apisix-custom-values.yaml apisix expertflow/apisix --version 5.1.0

  6. You’ve successfully enabled or disabled the authentication/authorization.

For resource_registration_endpoint and discovery settings:

  • Ensure that, instead of using the FQDN, the Keycloak service name is specified before the /auth subpath as shown above.

  • Ensure that your realm name or tenant ID is used instead of "<yourRealm>".

For details on secured components’ APIs and the access levels assigned to different roles, please refer to the Secured Components document.

IAM Configurations

  1. Go to your IAM admin console and select your realm.

  2. Click on Clients option in left panel and select cim.

 

  1. Go to the Client scopes and click on cim-dedicated under Assigned client scope.

 

  1. Click on Add Mapper and then select By configuration.

 

  1. Click on Audience.

 

  1. Provide name as “fix-audience”. Select cim in Included Client Audience. Check “Add to ID Token” toggle as On and Save.

 

Create Customer User

This user is responsible for creation of Access Token for each customer in Customer Widget.

  1. Go to the Users tab in left menu.

 

  1. Click on Add User. Provide username as “customer” and toggle Email Verified On. Click Create.


  1. Once user is created, go to Credentials and click on Set Password. Provide credentials as “customer” and toggle temporary to Off. Click Save.

 

  1. Click on the Realm roles and check if the customer role exists. If it exists then skip the creation process, else create a customer role.

 

  1. After creating customer user, assign it a customer role. Go to the customer user and the click on Role mapping.

 

  1. Click on Assing role and assign customer role to user along with offline_access and uma_authorization.

 

  1. Customer user creation is done.
